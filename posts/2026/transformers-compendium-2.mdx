## Overlapping disk reads with GPU transfers [#overlapping-disk-reads-with-GPU-transfers]

Transformers has two paths for loading weights, a synchronous and asynchronous version, designed for different loading scenarios.

<Snippet>
{`
# conceptual loop
model = ModelClass(config)              # empty skeleton, weights on "meta"
with safe_open("model.safetensors") as f:
    for name in f.keys():
        tensor = f.get_tensor(name)     # 1. read bytes off disk into CPU RAM
        tensor = tensor.to("cuda",      # 2. host → device copy over PCIe
                           dtype=torch.float16)
        set_module_tensor(model, name, tensor)  # 3. attach to the right param
`}
</Snippet>

The synchronous version uses a single thread to load a single weight at a time. This thread is blocked from doing anything else until the job is complete.

1. The thread enters the kernel which checks whether the raw tensor bytes are already in memory. If not, the kernel requests and moves the bytes from storage to memory. When the bytes are being moved, the thread and GPU are idle because they have to wait until all the bytes are transferred. Once they're in memory, the thread wakes up and copies and reads the bytes into a CPU tensor. The next line of code can be executed now.
2. PyTorch calls [cudaMemcpy](https://developer.download.nvidia.com/compute/DevZone/docs/html/C/doc/html/group__CUDART__MEMORY_g48efa06b81cc031b2aa6fdc2e9930741.html) to copy the bytes from the CPU to GPU memory over PCIe. Again, the thread is blocked from progressing until the bytes are finished copying over. After they're are in GPU memory, the thread copies and reads them into the new GPU tensor and the next line of Python runs.
3. Find and replace the matching parameter on the meta model with the actual tensor.

The thread waits in steps 1 and 2 while the bytes are being transferred from disk to CPU memory and from CPU memory to GPU memory.

This is useful in memory-constrained scenarios like quantization because if you have multiple threads, they may pull weights to the device faster than the main thread can quantize and release those weights.

<SyncAsyncLoadingDiagram />

The asynchronous version increases the number of threads to 4. Each thread can work on a job, disk read and copy from CPU to GPU, while the main thread continues going through the state dict and assembling the model.

But even if there are multiple threads, the GIL only allows one thread to execute at a time. Two threads doing work on the CPU takes the same amount of time as one thread doing the work sequentially because they're only taking turns holding the GIL.

PyTorch `.to(device)` releases the GIL during `cudaMemcpy` because it doesn't require the GIL for any CUDA work. This allows threads to exist and run in different stages. With the GIL released, thread A can copy bytes from CPU to GPU and then wait for it to finish. In the meantime, because thread A is blocked and doesn't need the GIL, it is free for thread B to pick up and do a disk read. When the first copy completes, thread A wakes up and gets the GIL back to wrap the resulting tensor in a Python object.

Threads and jobs are managed by [ThreadPoolExecutor](https://docs.python.org/3/library/concurrent.futures.html#threadpoolexecutor). It creates a pool of up to `max_workers` threads and reuses them instead of spawning a new one for each job.

For every weight, a job is submitted to the pool and stashed as a `Future`, a reference pointing to a function that hasn't been run yet. This detaches *submitting* work from *performing* the work because any free worker thread can execute the `Future` job and write the tensors into the model.

By the time the main thread gets to weight *N*, weights *N+1* to weights *N+k* are already being processed or done.

<WorkersVsLoadTimeChart />

While it may seem like increasing the number of workers would decrease loading time, after a certain point, extra workers stop being helpful because there is only a small number of shared physical resources.

There is the storage device, CPU memory for staging buffers, and PCIe copy to GPU. 2-3 workers are enough to keep disk and PCIe transfers busy plus a 4th worker to fill in any gaps. Hardware utilization is already close to 100%.

Adding more workers doesn't make it faster, but it will cost you. Many small frictions compound when the number of workers increases beyond saturation. The extra workers are competing for the same resources. For example, the GPU usually has one or two copy engines. With 16 workers making concurrent `cudaMemcpy` calls, the copy engine queues most of these calls which adds overhead.

In Transformers, 4 workers is enough to keep the disk and PCIe pipelines humming.

## CUDA caching allocator warmup [#CUDA-caching-allocator-warmup]

When PyTorch moves a tensor to the GPU, it uses a caching allocator on top of [cudaMalloc](https://developer.download.nvidia.com/compute/DevZone/docs/html/C/doc/html/group__CUDART__MEMORY_gc63ffd93e344b939d6399199d8b12fef.html), because otherwise, it would be very slow to assign memory on every call. It's slow because `cudaMalloc` is synchronous and it has to find a free region in GPU memory that's big enough.

1. The allocator requests `cudaMalloc` to assign a block of memory that's at least as large as tensor size N.
2. When the tensor is freed, the allocator holds on to that block in a cache.
3. The next request whose tensor size fits reuses that block directly from the cache. It doesn't make a separate `cudaMalloc` call.

The `cudaMalloc` cost is low overall over the loading process, but the first time the allocator requests a block of memory of a certain size, it is still slow. For a 70B model with many distinct tensor shapes, even with an allocator, this adds up and becomes a real bottleneck.

Transformers performs one big allocation upfront and then discards it.
<Snippet>
{`
for device, byte_count in total_byte_count.items():
    # ... memory accounting to pick the right byte_count ...
    # We divide by 2 here as we allocate in fp16
    _ = torch.empty(int(byte_count // 2), dtype=torch.float16,
                    device=device, requires_grad=False)
`}
</Snippet>

`torch.empty()` does two important things.

- Make one really large `cudaMalloc` call to the allocator. This single block is sized to the whole model.
- The GPU tensor is assigned to `_` and on the next iteration, `_` gets reassigned to something else. Since that tensor isn't being referenced anymore, the block is released back to the allocator.

<CudaAllocatorDiagram />

Crucially, when that tensor is freed, PyTorch doesn't return the block of memory to the driver. It keeps it for reuse, so this way, Transformers only pays the slow `cudaMalloc` cost once upfront. The throwaway `_` warms up the allocator by requesting memory ahead of time before the real weights are loaded. When a real weight is loaded, PyTorch uses this block without calling `cudaMalloc` again. Every subsequent `.to(device)` uses small pieces of memory from the large pre-fetched block.

Pre-allocating more memory than the model size doesn't help. After making the single big `cudaMalloc` call, subsequent loading is dominated by data movement and not available memory.
